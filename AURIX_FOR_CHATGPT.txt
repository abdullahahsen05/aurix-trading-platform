----------------------------------------------------------------
WHAT IS AURIX?
----------------------------------------------------------------
Aurix is a prop trading firm management platform. It is a web
application where traders connect their MetaTrader 5 (MT5) broker
accounts and get a full dashboard: performance analytics, equity
tracking, risk monitoring, and portfolio overview. Admins (the
firm's staff) oversee all traders from a separate admin workspace,
manage risk rules, handle CRM, and supervise accounts.

Think of it as a white-label back-office for a funded trader program
(like FTMO, MyForexFunds, etc.). Multiple traders use the platform
simultaneously under one firm's admin.

USER ROLES
----------------------------------------------------------------
Two roles: TRADER and ADMIN

TRADER:
- Can only access their own accounts and trades
- Sees trader-specific pages in sidebar
- Cannot access /admin/* routes

ADMIN:
- Can access all traders' data (bypasses RLS via service role)
- Sees admin-specific pages in sidebar
- Can also preview trader workspace

Role is stored in the `profiles` table (Supabase) alongside the
Supabase Auth user. Role-based access is enforced via:
  1. requireAuth() / requireAdmin() / requireTrader() in every API route
  2. assertCanAccessAccount() for account-level ownership checks
  3. Supabase RLS policies on every table

API ROUTES
----------------------------------------------------------------
All routes return ApiEnvelope<T>:
  Success: { ok: true, data: T }
  Failure: { ok: false, error: { code: string, message: string } }

Auth:
  GET  /api/auth/session                   Current user info
  PATCH /api/auth/profile                  Update profile (name, timezone)

Trading Accounts:
  GET  /api/trading-accounts               List accounts (trader: own, admin: all)
  POST /api/trading-accounts               Create new account
  GET  /api/trading-accounts/[id]          Account detail
  POST /api/trading-accounts/[id]/broker-credentials   Store encrypted MT5 creds

Trades:
  GET  /api/trades                         List trades (?accountId, ?status)
  POST /api/trader/sync-trades             Trigger broker sync (via MetaAPI)
  GET  /api/trader/daily-pnl              Today's closed P&L

Analytics:
  GET  /api/analytics/summary             AnalyticsSummary (?accountId, ?period)
  GET  /api/analytics/equity-curve        EquityPoint[] for chart (?accountId)

Risk:
  GET  /api/risk/rules                    List risk rules
  POST /api/risk/rules                    Create rule (admin only)
  PATCH /api/risk/rules/[id]             Update rule
  DELETE /api/risk/rules/[id]            Delete rule
  GET  /api/risk/events                   List unacknowledged events
  POST /api/risk/events/[id]/acknowledge  Mark event resolved

CRM:
  GET  /api/crm/traders                   Trader profiles list
  GET  /api/crm/notes                     Notes (?traderId)
  POST /api/crm/notes                     Create note

Admin:
  GET  /api/admin/summary                 Platform KPIs
  GET  /api/admin/users                   User list (paginated)
  PATCH /api/admin/users/[id]/status      Suspend/Activate user
  GET  /api/admin/accounts                All accounts
  PATCH /api/admin/accounts/[id]/status   Change account status
  GET  /api/admin/audit                   Audit log

Notifications:
  GET  /api/notifications                 User notifications
  POST /api/notifications/[id]/read       Mark read
  POST /api/notifications/read-all        Mark all read

Realtime:
  GET  /api/realtime/token               WebSocket auth token

BUSINESS LOGIC (src/lib/domain/metrics.ts + risk.ts)
----------------------------------------------------------------
Metrics calculated from raw trades array:
  calculateTotalProfit()       Sum of closed trade profits
  calculateWinRate()           % of profitable closed trades
  calculateMaxDrawdown()       Peak-to-trough equity % decline
  calculateRiskRewardRatio()   Avg win / avg loss
  calculateProfitFactor()      Gross profit / gross loss
  calculateConsistencyScore()  % of trading days that were profitable
  buildAnalyticsSummary()      Combines all into AnalyticsSummary DTO

Risk evaluation:
  evaluateRiskRules()          Checks if account breaches DAILY_LOSS,
                               MAX_DRAWDOWN, or OPEN_TRADES thresholds.
                               Returns array of RiskEventDto.
  shouldRestrictAccount()      Returns true if any CRITICAL severity event found.

REALTIME (current state)
----------------------------------------------------------------
Supabase Realtime:
  useRealtimeUpdates() hook subscribes to postgres_changes on:
  - account_snapshots   → invalidates account + equity-curve queries
  - trades             → invalidates trades + accounts queries
  - risk_events        → invalidates risk-events queries
  - notifications      → invalidates notifications queries

Socket.io:
  Server scaffolded in src/lib/realtime/server.ts
  Events defined in src/lib/realtime/events.ts:
    account.snapshot.updated
    trade.opened, trade.closed
    risk.event.created

IMPORTANT CONVENTIONS
----------------------------------------------------------------
1. API response format: always { ok: true, data } or { ok: false, error: { code, message } }
   Use helpers: jsonOk(data) and jsonFail(code, message, status)

2. Auth in API routes: always call requireAuth(), requireAdmin(),
   or requireTrader() at the top of every handler.

3. Supabase clients:
   - Browser: createClient() from lib/supabase/client.ts — RLS enforced
   - Server SSR: createClient() from lib/supabase/server.ts — cookie-based, RLS enforced
   - Admin bypass: createAdminClient() from lib/supabase/admin.ts — bypasses RLS

4. Data types: DB uses snake_case columns, DTOs use camelCase.
   All mapping happens in src/lib/mappers/.

5. Path alias: @/* maps to src/* (e.g. import from "@/lib/auth/session")

6. Mock data: all live data currently comes from src/lib/data/mockData.ts.
   Services are written to query Supabase but fall back to / are supplemented
   by mock data during development.

7. Next.js 16 note (from AGENTS.md): This version has breaking changes —
   APIs, conventions, and file structure may differ from training data.
   Read node_modules/next/dist/docs/ before writing any Next.js-specific code.
