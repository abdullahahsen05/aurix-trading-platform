PROJECT OVERVIEW
----------------
AURIX is a proprietary trading firm dashboard built with:
  - Next.js 16.2.6 (App Router, TypeScript)
  - React 19
  - Tailwind CSS 4
  - Zustand + TanStack React Query
  - Framer Motion, Radix UI, Lucide icons
  - Lightweight Charts (financial) + Recharts
  - Socket.io (real-time WebSocket layer, scaffolded but not live)
  - Prisma 7 ORM with a PostgreSQL schema (defined but no live DB connected)
  - Zod for validation, jose for JWT, bcryptjs for password hashing

The app is currently UI-only. All data comes from a mock data file.
There is no authentication, no database connection, and no live broker data.
The next step is to wire up Supabase as the database + auth provider.

DATABASE SCHEMA (prisma/schema.prisma)
================================================================================

Enums:
  UserRole:      TRADER | ADMIN
  UserStatus:    ACTIVE | SUSPENDED | PENDING
  AccountStatus: PENDING | CONNECTED | SYNCING | DISCONNECTED | RESTRICTED
  TradeStatus:   OPEN | CLOSED
  TradeSide:     BUY | SELL
  RiskSeverity:  INFO | WARNING | CRITICAL

Models:

  User
    id            cuid (PK)
    email         unique string
    passwordHash  string (nullable)
    name          string
    role          UserRole
    status        UserStatus (default PENDING)
    → has one TraderProfile
    → has many TradingAccount
    → has many AuditLog (as actor)

  TraderProfile
    id      cuid (PK)
    userId  unique FK → User
    segment string (default "EVALUATION")
    → has many CrmNote
    → has many CrmActivity
    → has many Subscription

  TradingAccount
    id              cuid (PK)
    userId          FK → User
    accountName     string
    brokerName      string
    brokerAccountId string (nullable)
    status          AccountStatus (default PENDING)
    currency        string (default "USD")
    → has one BrokerCredential
    → has many AccountSnapshot
    → has many Trade
    → has many DailyAccountMetric
    → has many RiskRule
    → has many RiskEvent
    → has many Notification
    Index: (userId, status)

  BrokerCredential
    id                  cuid (PK)
    tradingAccountId    unique FK → TradingAccount
    provider            string
    encryptedReference  string (stores encrypted MT5 credentials)

  AccountSnapshot
    id               cuid (PK)
    tradingAccountId FK → TradingAccount
    balance          Decimal(18,2)
    equity           Decimal(18,2)
    floatingPnl      Decimal(18,2)
    drawdownPercent  Decimal(7,2)
    capturedAt       DateTime
    Index: (tradingAccountId, capturedAt DESC)

  Trade
    id               cuid (PK)
    tradingAccountId FK → TradingAccount
    symbol           string (e.g. EURUSD, XAUUSD, NAS100)
    side             TradeSide
    status           TradeStatus
    volume           Decimal(12,4)
    openPrice        Decimal(18,6)
    closePrice       Decimal(18,6) nullable
    profit           Decimal(18,2)
    currency         string (default "USD")
    openedAt         DateTime
    closedAt         DateTime nullable
    Index: (tradingAccountId, status, openedAt DESC)

  DailyAccountMetric
    id                  cuid (PK)
    tradingAccountId    FK → TradingAccount
    metricDay           Date
    profit              Decimal(18,2)
    winRatePercent      Decimal(7,2)
    maxDrawdownPercent  Decimal(7,2)
    tradeCount          Int
    Unique: (tradingAccountId, metricDay)

  RiskRule
    id               cuid (PK)
    tradingAccountId FK → TradingAccount (nullable = platform-level rule)
    name             string
    severity         RiskSeverity
    metric           string (DAILY_LOSS | MAX_DRAWDOWN | OPEN_TRADES)
    threshold        Decimal(18,2)
    enabled          boolean

  RiskEvent
    id               cuid (PK)
    tradingAccountId FK → TradingAccount
    ruleName         string
    severity         RiskSeverity
    message          string
    acknowledgedAt   DateTime nullable
    createdAt        DateTime

  CrmNote
    id              cuid (PK)
    traderProfileId FK → TraderProfile
    authorName      string
    note            string
    createdAt       DateTime

  CrmActivity
    id              cuid (PK)
    traderProfileId FK → TraderProfile
    type            string
    description     string
    createdAt       DateTime

  Subscription
    id              cuid (PK)
    traderProfileId FK → TraderProfile
    planName        string
    status          string
    startedAt       DateTime
    endsAt          DateTime nullable

  Notification
    id               cuid (PK)
    tradingAccountId FK → TradingAccount (nullable)
    userId           string (nullable)
    title            string
    message          string
    readAt           DateTime nullable
    createdAt        DateTime

  AuditLog
    id          cuid (PK)
    actorUserId FK → User (nullable)
    action      string
    entityType  string
    entityId    string (nullable)
    metadata    Json (nullable)
    createdAt   DateTime

TYPESCRIPT DOMAIN TYPES (src/lib/domain/types.ts)
================================================================================

These are the DTOs used by all API routes and services:

  MoneyValue           { amount: number; currency: string }

  TraderAccountSummary {
    accountId, accountName, brokerName, status: AccountStatus,
    balance: MoneyValue, equity: MoneyValue, floatingPnl: MoneyValue,
    openTradeCount, drawdownPercent, updatedAt
  }

  TradeDto {
    id, accountId, symbol, side: TradeSide, status: TradeStatus,
    volume, openPrice, closePrice | null,
    profit: MoneyValue, openedAt, closedAt | null
  }

  AnalyticsSummary {
    accountId, totalProfit: MoneyValue, winRatePercent, maxDrawdownPercent,
    riskRewardRatio, consistencyScore, tradeCount,
    period: "DAILY" | "WEEKLY" | "MONTHLY" | "ALL_TIME"
  }

  EquityPoint          { capturedAt, balance, equity }

  RiskRuleDto {
    id, scope: "PLATFORM" | "ACCOUNT", name, severity: RiskSeverity,
    metric: "DAILY_LOSS" | "MAX_DRAWDOWN" | "OPEN_TRADES",
    threshold, enabled
  }

  RiskEventDto         { id, accountId, ruleName, severity, message, createdAt }

  CrmNoteDto           { id, traderId, authorName, note, createdAt }

  TraderProfileDto {
    traderId, name, email,
    segment: "EVALUATION" | "FUNDED" | "AT_RISK" | "VIP",
    accountCount, totalEquity: MoneyValue, lastActivityAt
  }

  AdminSummaryDto {
    activeTraders, connectedAccounts, openRiskEvents,
    monthlyRecurringRevenue: MoneyValue
  }

  ApiEnvelope<T>       { ok: true; data: T } | { ok: false; error: { code, message } }

================================================================================
DOMAIN LOGIC (src/lib/domain/)
================================================================================

metrics.ts — Pure calculation functions (no DB dependency):
  calculateTotalProfit(trades)       → sums closed trade profit
  calculateWinRate(trades)           → % of profitable closed trades
  calculateMaxDrawdown(equityCurve)  → peak-to-trough equity drawdown %
  calculateRiskRewardRatio(trades)   → avg win / avg loss
  calculateProfitFactor(trades)      → gross profit / gross loss
  calculateConsistencyScore(trades)  → % of trading days that were profitable
  buildAnalyticsSummary(...)         → combines all of the above into AnalyticsSummary

risk.ts — evaluateRiskRules():
  Takes { account, trades, rules, dailyProfit } and returns triggered RiskEventDto[]
  Checks DAILY_LOSS, MAX_DRAWDOWN, OPEN_TRADES thresholds

WHAT NEEDS TO BE BUILT (BACKEND + SUPABASE INTEGRATION)
================================================================================

1. SUPABASE SETUP
   - Replace Prisma PostgreSQL connection with Supabase project URL + anon key
   - Set DATABASE_URL to Supabase connection string in .env.local
   - Run prisma migrate dev to push schema to Supabase
   - Keep Prisma as the ORM (it works fine on top of Supabase Postgres)

2. AUTHENTICATION (replace mock session)
   - Use Supabase Auth (email/password) for login, register, forgot/reset password
   - On login: call supabase.auth.signInWithPassword()
   - On register: call supabase.auth.signUp(), then create User + TraderProfile rows
   - Session: use Supabase session tokens, store in cookies (SSR-safe)
   - Replace GET /api/auth/session to read from Supabase session cookie
   - Add middleware (src/middleware.ts) to protect trader/* and admin/* routes
   - Role-based access: read role from User table, redirect accordingly

3. SERVICE LAYER (replace all mock data with real DB queries via Prisma)
   Each service function should query Supabase PostgreSQL through Prisma client:

   tradeService.ts
     listTrades({ accountId, status })
       → prisma.trade.findMany({ where: { tradingAccountId, status }, orderBy: openedAt DESC })

   tradingAccountService.ts
     listTradingAccounts(userId)
       → prisma.tradingAccount.findMany({ where: { userId } })
     getTradingAccount(accountId, userId)
       → prisma.tradingAccount.findFirst({ where: { id, userId } })

   analyticsService.ts
     getAnalyticsSummary(accountId)
       → query trades + accountSnapshots from DB, then call buildAnalyticsSummary()
     getEquityCurve(accountId)
       → prisma.accountSnapshot.findMany({ where: { tradingAccountId }, orderBy: capturedAt ASC })

   riskService.ts
     listRiskRules(accountId?)
       → prisma.riskRule.findMany({ where: { OR: [{ tradingAccountId: null }, { tradingAccountId }] } })
     listRiskEvents(accountId?)
       → prisma.riskEvent.findMany({ where: { tradingAccountId }, orderBy: createdAt DESC })

   crmService.ts
     listTraderProfiles()    → prisma.traderProfile.findMany({ include: { user, tradingAccounts } })
     listCrmNotes(traderId?) → prisma.crmNote.findMany({ where: { traderProfileId } })
     createCrmNote(...)      → prisma.crmNote.create(...)

   adminService.ts
     getAdminSummary()
       → aggregate queries: count(User where active), count(TradingAccount where connected), etc.

4. API ROUTES (add auth guards, wire to real services)
   - All routes should verify Supabase session token from cookie before responding
   - Add POST /api/trading-accounts to create a new account
   - Add POST /api/crm/notes to create a CRM note
   - Add PATCH /api/risk/rules/[id] to toggle or update a rule
   - Add POST /api/admin/users/[id]/status to suspend/activate users

5. REAL-TIME (Socket.io + Supabase Realtime alternative)
   Option A: Keep Socket.io server, emit events from API route handlers after DB writes
   Option B: Replace with Supabase Realtime (postgres_changes subscriptions) on client
   Recommended for MVP: Supabase Realtime on the client for account snapshots + risk events

6. BROKER INTEGRATION (MetaApi - later phase)
   - Complete MetaApiBrokerAdapter.fetchSnapshot() using MetaApi SDK
   - Store encrypted MT5 credentials in BrokerCredential table
   - Create a background job / cron that polls MetaApi every 30s and upserts
     AccountSnapshot rows and Trade rows into Supabase
   - Emit account.snapshot.updated via Socket.io or Supabase Realtime after each poll
